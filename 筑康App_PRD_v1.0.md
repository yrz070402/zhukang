# 筑康 App 产品需求文档 (PRD)

**版本**: v1.0
**日期**: 2026-04-09
**产品名称**: 筑康 - 饮食、保健品与医疗一体化健康管理平台

---

## 1. 产品概述

### 1.1 产品愿景
筑康旨在帮助用户重新了解自己的身体，精准掌握每天的营养素摄入，恢复对健康的控制权。通过智能图像识别技术与个性化健康建议，为用户提供科学、简便的饮食管理工具，实现营养均衡、身体健康的长期目标。

### 1.2 核心理念
**"健康的生活习惯胜过一切治疗手段，助康，筑康，祝康"**

### 1.3 目标用户
- 关注健康的普通用户
- 健身人群（增肌/减脂）
- 有特殊饮食需求的人群（慢病患者、亚健康人群）
- 追求营养均衡的都市白领

### 1.4 产品形态
- **首发**: Android App（独立应用，完整功能体验）
- **后期**: 微信小程序（轻量化入口）+ iOS App

---

## 2. 市场背景与问题分析

### 2.1 市场痛点
| 痛点 | 描述 |
|------|------|
| 饮食失控 | 快餐与加工食品文化盛行，用户失去对健康的控制权 |
| 营养失衡 | 高糖高脂饮食、过度加工食物导致慢性病攀升 |
| 营养认知不足 | 即使看似均衡的饮食，也难以满足最佳营养需求 |
| 记录繁琐 | 现有健康App需要手动输入，使用门槛高 |

### 2.2 市场机会
- 健康意识觉醒，用户愿意为健康管理付费
- AI图像识别技术成熟，可降低使用门槛
- 健身风潮兴起，核心用户付费意愿强

### 3.2 核心功能详细设计

#### 3.2.1 饭前扫一扫 - 三餐营养素一键分析

**功能描述**：用户拍摄食物照片，系统自动识别菜品和食材，分析营养素含量。

| 要素 | 说明 |
|------|------|
| 输入 | 食物照片（拍照/相册选择） |
| 处理 | AI图像识别（豆包API）+ 中国食物成分表数据库 |
| 输出 | 蛋白质、脂肪、碳水化合物、微量营养素（钙、铁、维生素等）含量 |
| 响应时间 | < 5秒 |

**用户故事**：
> 作为一名关注健康的用户，我希望在吃饭前拍照就能知道这顿饭的营养成分，这样我可以轻松掌握自己的营养摄入状况。

**界面原型要点**：
- 拍照按钮置于首页显眼位置
- 分析结果以卡片形式展示
- 显示"已摄入占每日推荐比例"

**示例输出**：
```
"今天已摄入 40% 的蛋白质推荐量，钙质缺口约 500mg，建议晚餐添加一份牛奶或深绿色叶菜。"
```

---

#### 3.2.2 每日营养素进度条

**功能描述**：将营养目标细化为每日摄入指标，动态生成可视化进度条。

**进度条分类**：

| 类别 | 包含营养素 |
|------|-----------|
| 宏量营养素 | 蛋白质、碳水化合物、脂肪 |
| 微量营养素 | 钙、铁、维生素D等 |
| 其他健康指标 | 膳食纤维、钠盐摄入量 |

**状态显示规则**：

| 颜色 | 含义 | 阈值 |
|------|------|------|
| 🟢 绿色 | 达标 | 80%-100% |
| 🟡 黄色 | 即将超标/不足 | 100%-120% 或 50%-80% |
| 🔴 红色 | 严重不足/过量 | <50% 或 >120% |

**用户故事**：
> 作为一名用户，我希望通过直观的进度条了解今天的营养状况，这样我可以快速知道晚餐该怎么调整。

---

#### 3.2.3 个性化建议与改善方案

**功能描述**：基于用户饮食数据，提供食材改进方案和补充剂推荐。

**子功能模块**：

**A. 智能饮食分析与建议**
- 分析已摄入营养素，识别缺失项
- 优先推荐天然食材补充方案
- 次选推荐保健品（附临床研究数据和医生背书）

**B. 个性化健身饮食计划**

| 目标 | 饮食特点 |
|------|---------|
| 增肌 | 高蛋白（鸡胸肉、蛋白粉等） |
| 减脂 | 低碳水、高纤维（绿叶蔬菜、瘦肉等） |

**C. 餐饮品牌合作**
- 与美团、饿了么合作
- 或推出自有健康餐品牌
- 提供定制化健康餐配送服务

**用户故事**：
> 作为一名健身爱好者，我希望系统根据我的增肌目标推荐合适的饮食方案，这样我可以更科学地达成目标。

---

#### 3.2.4 突发事件记录与调整

**功能描述**：用户记录突发事件（过度饮食、饮酒、失眠等），系统自动调整营养目标。

**支持的突发事件类型**：
- 过度饮食
- 饮酒
- 失眠
- 其他（用户自定义）

**系统响应**：
- 自动调整营养素目标
- 生成针对性饮食调整建议
- 显示"不良影响进度条"，展示恢复程度

**示例输出**：
```
"过度饮食后的影响已缓解 20%。建议：增加膳食纤维摄入，多喝水，避免高脂食物。"
```

---

#### 3.2.5 数据报告

**功能描述**：提供多维度健康数据报告。

| 报告类型 | 内容 | 频率 |
|---------|------|------|
| 每日报告 | 当天营养完成情况、亮点与不足 | 每日生成 |
| 周报 | 一周饮食趋势、长期问题标记 | 每周一 |
| 月报 | 月度饮食分析、体重变化、健康风险预测 | 每月1日 |

**数据可视化**：
- 饼图：营养素占比
- 柱状图：每日摄入趋势
- 折线图：长期变化趋势

---

#### 3.2.6 微信状态与社交激励

**功能描述**：与微信生态打通，增加社交趣味性。

**功能点**：
- **微信状态自动更新**：完成每日营养目标时自动触发（如"今天也好好吃饭"）
- **健康积分排行榜**：类似微信步数，邀请好友比拼
- **打卡激励**：获得称号和积分

**用户故事**：
> 作为一名用户，我希望完成健康目标后能分享到微信状态，这样可以激励自己坚持健康饮食。

---

#### 3.2.7 健身人群专属功能

**功能描述**：为健身人群提供精准饮食支持。

**功能模块**：

| 模块 | 说明 |
|------|------|
| 增肌计划 | 高蛋白优先，结合运动量动态调整卡路里 |
| 减脂计划 | 低碳水、高纤维餐单推荐 |
| 运动数据同步 | 接入Keep、小米运动等平台 |

**数据同步内容**：
- 卡路里消耗
- 运动强度
- 运动时长

---

#### 3.2.8 健康生态闭环

**功能描述**：打造从饮食到医疗的健康管理生态。

**生态组成部分**：
- **智能设备整合**：体脂秤、血糖仪等数据接入
- **家庭医生服务**：医生直接给出营养素量化建议
- **健康险推荐**：良好健康记录用户享受保费减免

## 5. 技术实现方案

### 5.1 技术架构

```
┌─────────────────────────────────────────────────────┐
│                    客户端层                          │
├─────────────────────────────────────────────────────┤
│  Android App (Java/Kotlin) ← 首发平台               │
│  微信小程序 / iOS App (后期扩展)                     │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│                    API 网关层                        │
│  FastAPI + CORS                                     │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│                    服务层                            │
├──────────────────┬──────────────────┬───────────────┤
│ 食物分析服务      │ 营养计算服务      │ 用户服务       │
│ food_analysis.py │ nutrition_calculator.py │ user.py │
└──────────────────┴──────────────────┴───────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│                    外部服务层                        │
├──────────────────┬──────────────────┬───────────────┤
│ 豆包 API (AI识别) │ 中国食物成分表     │ 运动平台API   │
│  ✓ 已确认        │ 营养数据库         │ (Keep等)      │
└──────────────────┴──────────────────┴───────────────┘
```

### 5.2 核心API接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/food/analyze` | POST | 上传食物图片，返回营养分析 |
| `/api/v1/nutrition/recommendation` | POST | 获取个性化营养建议 |
| `/api/v1/user/profile` | GET/PUT | 用户档案管理 |
| `/api/v1/report/daily` | GET | 获取每日报告 |
| `/api/v1/report/weekly` | GET | 获取周报 |
| `/api/v1/report/monthly` | GET | 获取月报 |

### 5.3 数据存储

| 数据类型 | 存储方案 | 说明 |
|---------|---------|------|
| 用户数据 | PostgreSQL | 结构化数据 |
| 饮食记录 | PostgreSQL | 支持复杂查询 |
| 营养数据库 | PostgreSQL + 缓存 | 中国食物成分表 |
| 图片文件 | 对象存储 (OSS) | 食物照片存储 |

## 9. 风险评估

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| AI识别准确率不足 | 用户信任度下降 | 人工标注+模型持续优化 |
| 数据隐私问题 | 法律风险、用户流失 | 符合GDPR/个人信息保护法 |
| API成本过高 | 利润压缩 | 本地缓存+批量处理 |
| 竞品模仿 | 市场份额下降 | 快速迭代+差异化功能 |
